## What this is

**Bahana Stress Tester** is a scoped fork of PortfolioOptimizer for Bahana TCW: a Streamlit web app + Python library covering **Portfolio Input + Stress Testing + Risk Analytics** — multi-source price ingestion, historical crisis replay (actual per-stock returns with beta-scaled proxy fallback), DCC-GARCH/Student-t-copula/HMM-regime sector shock propagation, Leontief macro contagion, Monte Carlo simulation, and a baseline VaR/Sharpe/GARCH risk dashboard. It has meaningful Indonesian-market (IDX, `.JK` tickers) support alongside US equities. Built for a solo/small-team quant workflow, not a regulated production trading system — treat it as a calculation-assistance tool (see README.md's disclaimer).

Optimization, Portfolio Builder (ranking/correlation-network), Factor Analysis, Reports, and Stock Valuation are **out of scope** and have been removed from this fork — see `docs/BUILD_SPEC.md` Phase 0/1 for the dependency audit that confirmed what could be deleted cleanly. Risk Analytics was originally a documented Phase 6 fast-follow with its backing modules kept dormant; Phase 6 has since restored `3_Risk_Analytics.py` against them (see below) — they are active, not dormant, as of this version.

## Load-bearing decisions

- **`src/risk/sector_beta.py` is load-bearing from two different call sites** — `SectorStressEngine.fit()` (`sector_stress.py`) uses `SectorBetaAnalyzer.compute()` for the cross-sector beta matrix shown in the UI, while the actual per-stock betas used for P&L come from `compute_all_stock_betas()` in `stock_sector_beta.py`. Separately, `MacroStressEngine.fit()` (`macro_stress.py`) lazily imports `SectorBetaAnalyzer` again for sector-return construction (Step 3 of its fit) — a real, core-path usage, not a fallback. Neither `sector_beta.py` nor `stock_sector_beta.py` can be merged or deleted. `sector_beta.py`'s own docstring on `compute_stock_betas_vs_portfolio_sectors()` used to claim it was the primary path `SectorStressEngine.fit()` uses — that claim was wrong (the method is called nowhere outside its own definition) and has been corrected (`docs/BUILD_SPEC.md` Phase 3 item 1, done).
- **Two package `__init__.py` landmines exist in this codebase's history — watch for a third.** Deleting a module that a sibling module's package `__init__.py` still imports breaks *every* consumer of that package, not just the deleted module's own callers (Python runs a package's `__init__.py` on any submodule import). This is exactly how `src/portfolio/__init__.py` would have broken Portfolio Input if `calculator.py`/`rebalancer.py` had been deleted without also trimming `__init__.py`'s imports (now fixed — `__init__.py` only exports `holdings.py` names). `src/risk/__init__.py` imports from `metrics.py`/`var.py`/`garch.py`, but since those three files were kept in place throughout (dormant through Phase 1-5, active as of Phase 6 — never deleted), that package's `__init__.py` never needed a change either way. Before deleting any module in a package with siblings, check the package's `__init__.py` first.
- **Preset loading populates `current_holdings` via a live price fetch, not just `weights`/`portfolio_value`** — `src/utils/preset_manager.py::apply_preset_to_state()` only ever sets `tickers`/`weights`/`portfolio_value`/`settings`/`loaded_preset_id`; it deliberately still does not touch `current_holdings`. `app/pages/1_Portfolio_Input.py`'s `_load_preset_and_populate_holdings()` wraps it and additionally derives `shares = weight * portfolio_value / current_price` — kept out of `preset_manager.py` on purpose to keep that module free of a network/`DataManager` dependency.
- **Portfolio Presets is a tab on Portfolio Input, not a separate page** — merged in because preset-loading and holdings-editing are the same workflow.
- **Correlation Network's node coloring reuses `network.py::node_color_for_percentile()` unmodified, but feeds it per-ticker P&L percentile instead of the Portfolio Builder composite-ranking percentile it was originally written for** — same tri-tier function, different input dimension entirely; this was an explicit design instruction (Phase 7a), not a coincidental reuse.
- **Correlation Network's P&L source is deliberately limited to Historical and Sector Shock scenario results — Macro Contagion is never offered.** `MacroStressEngine.run_stress()` (`macro_stress.py`) computes `direct`/`total` return per-ticker by looking up `contagion.h_initial`/`contagion.leontief_total` keyed on **sector only** (`macro_stress.py`, the `for ticker, weight in weights.items()` loop) — every ticker sharing a sector gets an identical return, scaled only by that ticker's own weight for the dollar P&L. That's real position-sized P&L, but not real per-ticker *differentiation* — coloring nodes by it would visually imply a per-ticker signal that doesn't exist. Don't add Macro Contagion as a Correlation Network P&L source without first adding a real ticker-level allocation step (e.g. ticker's sector-beta × its sector's distress, the same shape Sector Shock already uses).
- **`MarketRegimeDetector.get_regime_correlation()` returns only a DataFrame — no flag distinguishing a real averaged matrix from its sub-5-observation identity-matrix fallback.** The Correlation Network tab's regime overlay (Phase 7b) needs to warn the user before that fallback renders as if it were a real network, but modifying that function is against this project's absolute constraints. Instead, `2_Stress_Testing.py`'s `_n_common_regime_obs()` read-only replicates the function's own aligned-observation count (same date-intersection between `regime_result.state_sequence` and `dcc_result.conditional_volatilities.index`) to decide whether to show the warning, then still calls the real function afterward to get the actual matrix. Verified bit-exact against the real function on both a normal-history case and a manufactured 3-observation case — if you touch either side of this duplication, re-verify they still agree.
- **Both Correlation Network sections (7a ticker, 7b sector-regime) render through one shared `_render_correlation_network()` helper in `2_Stress_Testing.py` — edge hover and edge color both work differently than they first appear.** (1) Edge hover: a Plotly `mode="lines"` trace only matches hover near its actual plotted points (the two edge endpoints), not along the interior of the segment — hovering the middle of a long edge shows nothing unless something else is plotted there. The fix is an additional, separate `go.Scatter` trace of invisible (`opacity=0`) markers, one per edge, positioned at each edge's exact midpoint and carrying the `ρ=` hover text — this is why the function builds `edge_traces` (visible lines, `hoverinfo="skip"`) and a single combined `edge_hover_trace` (invisible markers, `hoverinfo="text"`) separately, rather than putting hover text on the lines themselves. That same invisible-marker trace also carries the visible colorbar legend (`marker.colorscale`/`cmin=-1`/`cmax=1`/`colorbar=...`), since a colorbar has to be attached to a trace with a real colorscale mapping, not to individually-colored line traces. (2) Edge color: `_edge_color_gradient()`, a page-local function using `plotly.colors.sample_colorscale("Turbo", ...)`, not `network.py`'s `edge_color_for_correlation()` — Turbo instead of a 2-flat-color or red-white-blue diverging scale, because a diverging scale washes out to near-white around correlation ≈ 0, making weak correlations hard to distinguish; Turbo stays vivid and discriminable across the whole `[-1, 1]` range. `network.py`'s own function was left untouched (still importable, just unused by this page) rather than modified, consistent with not rewriting a reviewed/merged module's function as a side effect of a page-level styling change. (3) All edges are solid lines — no dashing anywhere, per explicit user request; MST/backbone-vs-additional-threshold-edge is still conveyed by line **width** (thick vs thin), just not by dash style. (4) 7b passes `mst_edges=set(graph.edges())` (i.e., every edge) into this shared function rather than a real MST subset, since it now renders a complete graph — every edge gets the "is_mst" thick-width styling and none get the thin one, which is intentional (there's no more MST-vs-extra distinction to convey once every pair is drawn).

## Known placeholders — do not mistake these for real, load-bearing values

- **`src/data/lseg_sectors.py::IDX_SECTOR_OVERRIDES["TLKM.JK"]`**: applied unconditionally in code, but carries an inline `# Confirm vs LSEG assignment` comment — i.e. shipped before the override was actually confirmed correct. - **`src/risk/sector_beta.py::compute_stock_betas_vs_portfolio_sectors()` is NOT on `SectorStressEngine.fit()`'s path** — grep-verified: it's called nowhere in the codebase outside its own definition. Its docstring used to claim otherwise (fixed, `docs/BUILD_SPEC.md` Phase 3 item 1); the real per-stock-beta path is `compute_all_stock_betas()` in `src/risk/stock_sector_beta.py`.
- **`scenarios.py`'s legacy uniform-shock dict is `UNIFORM_SHOCK_SCENARIOS`; `historical_scenarios.py`'s actual-per-stock-returns list is `PER_STOCK_CRISIS_SCENARIOS`** — previously both were named `HISTORICAL_SCENARIOS` (a naming collision fixed in `docs/BUILD_SPEC.md` Phase 3 item 2). Both are still live and used for different things — don't conflate them.
- **`sector_stress.py` reporting `converged=False` for a fit `dcc_garch.py` logged as `converged=True`** — a known, not-yet-fixed bookkeeping bug. `docs/BUILD_SPEC.md` Phase 2 item 1 — deferred, not yet addressed in this fork.
- **`3_Risk_Analytics.py`'s EWMA Volatility panel used to plot the wrong series** — it computed `ewma_volatility(returns, decay=0.94)` but then plotted `(returns*weights).sum(axis=1).rolling(20).std()` instead of that computed series (a pre-fork, pre-existing bug). Fixed on restoration (Phase 6): the panel now computes EWMA vol directly on the portfolio return series and plots that.
- **`src/portfolio_builder/network.py`'s module docstring used to justify staying at ticker-level plain `.corr()` (instead of extending `DCCGARCHModel` to ticker level) by citing "an unresolved convergence-misreport issue"** attributed to the since-deleted `fetch.py`. That claim never traced to anything in `dcc_garch.py`'s actual code — what's really there is a hard `ConvergenceError` guard plus an explicit constant-vol fallback (see `docs/architecture.md`'s `src/risk/` section). Fixed on activation (Phase 7a): the docstring now describes the real mechanism and no longer cites the unverified reason. The underlying decision (plain `.corr()` at ticker level) was never wrong — only the stated justification was.

## Absolute constraints

- Streamlit + the existing Python backend only. No React, Redis, graph database, WebSockets, or 3D rendering anywhere in this codebase (confirmed: no `package.json`, no such dependencies in `requirements.txt`, no matching imports).
- Do not modify the existing DCC-GARCH or sector-normalization modules' core algorithms as a side effect of unrelated work — extend via new config fields/functions, not by rewriting what's already reviewed and merged.
- Do not re-import from a deleted module (`src/optimization/`, `src/factors/`, `src/reports/`, `src/valuation/`, `src/portfolio/{calculator,rebalancer}.py`, `src/portfolio_builder/{fetch,ranking,metrics,heat_color,ff5_overlay}.py`) or a deleted page (`2_Optimization.py`, `5_Monitoring.py`, `6_Factor_Analysis.py`, `7_Reports.py`, `8_Stock_Valuation.py`, `10_Portfolio_Builder.py`) — these are gone, not just unlinked from nav. (`3_Risk_Analytics.py` was restored in Phase 6 — it is no longer on this deleted list.)
- Python 3.11 or 3.12 only (pinned in `runtime.txt`) — required for `hmmlearn` wheel availability on Streamlit Community Cloud; the Sector Shock stress test's HMM regime-conditioning does not run on other versions.
